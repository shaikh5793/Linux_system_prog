Position-independent code (PIC) is a type of machine code that can be
executed at any memory location without modification. It is typically
used in shared libraries, where the same library code may be loaded into
memory at different locations for different processes. PIC code is able
to execute correctly no matter where it is loaded in memory because it
uses relative addressing instead of absolute addressing.

PIC code is typically generated by using the -fPIC option when compiling
the code.

The main advantages of PIC are:

It allows for shared libraries to be loaded at different memory
locations for different processes, which reduces the memory footprint of
the system. PIC code can be loaded into memory at any location and can
be executed without modification, which makes it more robust and less
prone to errors. It also allows for shared libraries to be loaded and
unloaded dynamically, which is useful for applications that use plug-ins
or dynamically loaded modules. It is also required in certain scenarios,
for example:

When a program is executed in a PIE (Position Independent Executable)
format, which can be loaded at any address When the same code is used by
multiple processes, it is loaded at different memory addresses, PIC
ensures that the code works correctly even when it is loaded at a
different address. PIC is also required for shared libraries, which are
loaded at different addresses in different processes. Please note that
PIC has a slightly larger memory footprint and may have a small
performance penalty compare to position dependent code (non-PIC) due to
the added complexity in the addressing mechanism.
